Campos Raiz

Para comear, temos algumas opes na raiz do TSConfig - essas opes so relacionadas a como seu projeto TypeScript ou JavaScript est configurado.

#Incluso de arquivos

Estas configuraes garantem que o Typescript pegue os arquivos corretos.

# Excluir - exclude

Especifica uma array de nomes de arquivos ou padres que devem ser ignorando durante o include.

Importante: exclude altera apenas os arquivos que esto nos resultados da configurao include. Um arquivo marcado como exclude ainda pode fazer parte do seu cdigo atravs de uma instruo import, uma incluso de types, uma diretiva /// <reference ou sendo relacionado na lista de files.

No um mecanismo que impede um arquivo ser includo no cdigo base - apenas altera o que a configurao include pode selecionar.

  • Default:

    ["node_modules", "bower_components", "jspm_packages"], plus the value of outDir if one is specified.

  • Related:

    include, files

# Estender - extends

O valor de extends uma string que contm o caminho para outro arquivo de configurao do qual ele herdar. O caminho pode usar o estilo de resoluo do Node.Js

A configurao do aquivo base carregada primeiro, e ento sobrescrita por aquelas presentes no arquivo de configurao herdado. Todos os caminhos relativos encontrados no arquivo de configurao sero resolvidos relativamente ao arquivo de configurao em que estes se originaram.

Vale notar que files, include e exclude da configurao que est sendo estendida sobrescreve aquelas definidas na configurao base, e circularidade entre arquivos de configurao no permitida.

Atualmente, a nica propriedade de top-level que excluda da herana references.

Exemplo

configs/base.json:

{
  "compilerOptions": {
    "noImplicitAny": true,
    "strictNullChecks": true
  }
}

tsconfig.json:

{
  "extends": "./configs/base",
  "files": ["main.ts", "supplemental.ts"]
}

tsconfig.nostrictnull.json:

{
  "extends": "./tsconfig",
  "compilerOptions": {
    "strictNullChecks": false
  }
}

Propriedades com caminhos relativos encontrados no arquivo de configurao, e que no sejam excludas da herana, sero resolvidos relativamente ao arquivo de configurao em que estas se originaram.

  • Default:

    false

  • Released:

    2.1

# Arquivos - files

Especifica uma lista de arquivos permitidos para incluir no programa. Um erro ocorre se qualquer um dos arquivos no puder ser encontrado.

    "sys.ts",
    "types.ts",
    "scanner.ts",
    "parser.ts",
    "utilities.ts",

Esta opo til quando voc tem uma quantia pequena de arquivos e no precisa utilizar um glob para se referir a diversos arquivos. Se voc precisa utilizar glob, ento use include.

    false

  • Related:

    include, exclude

# Incluir - include

Especifica um array de nomes de arquivos ou padres a serem includos no programa. Esses nomes de arquivo so resolvidos em relao ao diretrio que contm o arquivo tsconfig.json.

json
{
  "include": ["src/**/*", "tests/**/*"]
}

Isso incluir:

include e exclude suporta caracteres curinga para criar padres globais:

  • ? corresponde a qualquer caractere (excluindo separadores de diretrio)
  • **/ corresponde a qualquer diretrio aninhado a qualquer nvel

Se um padro global no incluir uma extenso de arquivo, apenas os arquivos com extenses compatveis sero includos (por exemplo: .ts, .tsx, e .d.ts por padro, com .js e .jsx se allowJs for definido como verdadeiro).

    [] if files is specified, otherwise ["**/*"]

    2.0

# Aquisio de tipo - typeAcquisition

Quando voc tem um projeto JavaScript no seu editor, TypeScript providenciar tipos para o seu node_modules automaticamente utilizando o conjunto de definies de @types, DefinitelyTyped. Isso conhecido como aquisio de tipo automtica e voc pode modific-la utilizando o objecto typeAcquisition no seu arquivo de configurao.

Se voc quiser desabilitar ou modificar essa funcionalidade, crie o jsconfig.json na raz do seu projeto:

json
{
  "typeAcquisition": {
    "enable": false
  }
}

Se voc quer incluir um mdulo especfico (mas ele no est em node_modules):

json
{
  "typeAcquisition": {
    "include": ["jest"]
  }
}

Se um mdulo no deve ser automaticamente adquirido, por exemplo, se uma biblioteca est disponvel no seu node_modules, mas a sua equipe concordou em no utiliz-lo:

json
{
  "typeAcquisition": {
    "exclude": ["jquery"]
  }
}
  • Default:

    false

#Opes de Projeto

Essas configuraes so usadas para definir as expectativas de tempo de execuo, como e onde voc deseja que o JavaScript seja emitido e o nvel de integrao que voc deseja com o cdigo JavaScript existente no seu projeto.

# Permitir JS - allowJs

Permite que arquivos JavaScript sejam importados dentro do seu projeto ao invs de s permitir arquivos .ts e .tsx. Por exemplo, este arquivo JS:

// @filename: carta.js
export const naipePadrao = "Copas";Try

Quando importado em um arquivo TypeScript, vai emitir um erro:

// @filename: index.ts
import { naipePadrao } from "./carta";

console.log(naipePadrao);Try

Mas importado normalmente com a opo allowJS ativada:

// @filename: index.ts
import { naipePadrao } from "./carta";

console.log(naipePadrao);Try

Esta opo pode ser utilizada como uma forma de migrar um projeto JavaScript para TypeScript de forma incremental. Permitindo que arquivos .ts e .tsx coexistam no mesmo projeto que os arquivos JavaScript.

  • Default:

    false

  • Related:

    checkJs, emitDeclarationOnly

  • Released:

    1.8

# Checar JS - checkJs

Funciona em conjunto com o allowJs. Quando o checkJs est ativado, os erros so relatados tambm nos arquivos JavaScript. Isto o equivalente a incluir // @ts-check na parte superior de todos os arquivos JavaScript includos no seu projeto.

Por exemplo, este um JavaScript incorreto, de acordo com a definio do tipo parseFloat que acompanha o TypeScript:

js

Quando importado em um mdulo TypeScript:

// @filename: index.ts

console.log(pi);Try

Voc no vai obter nenhum erro. No entanto, se voc ativar o checkJs ento voce tambm ter mensagens de erro no seu arquivo JavaScript.

    false

  • Related:

  • Todos os arquivos de implementao devem corresponder a um padro do include ou listados no array files. Se esta restrio for violada, o tsc informar quais arquivos no foram especificados.
  • declaration setado como true

Voc pode encontrar a documentao de projetos TypeScript no guia.

# Declaraes - declaration

Gere arquivos .d.ts para cada arquivo TypeScript ou JavaScript dentro do seu projeto. Esses arquivos .d.ts so arquivos de definio de tipo que descrevem a API externa do seu mdulo. Com arquivos .d.ts, ferramentas como o TypeScript podem fornecer intellisense e tipos mais precisos para cdigo que ainda no foi digitado.

Quando a opo declaration definida como true, executando o compilador com este cdigo TypeScript:

export let olaMundo = "ol!";Try

Vai gerar um arquivo index como este:

export let olaMundo = "ol!";Try

Com um outro arquivo correspondente olaMundo.d.ts:

export declare let olaMundo: string;Try

Ao trabalhar com arquivos .d.ts para arquivos JavaScript, voc pode usar emitDeclarationOnly ou usar outDir para garantir que os arquivos JavaScript no sejam sobrescritos.

  • Default:

    false

  • Related:

    declarationDir, emitDeclarationOnly

  • Released:

    1.0

Voc deve fortemente considerar ativar essa opo se estiver usando referncias de projeto.

# Iterao Downlevel - downlevelIteration

Downleveling o termo do TypeScript para transpilar para uma verso mais antiga do JavaScript. Esse sinalizador permite que, em runtimes mais antigos do JavaScript, haja o suporte a uma implementao mais precisa de como o JavaScript moderno interage com novos conceitos.

O ECMAScript 6 adicionou vrias novas primitivas de iterao: o loop for / of (for (el of arr)), operador de spread ([a, ...b]), spread de argumento (fn (... args)) e o Symbol.iterator. --downlevelIteration permite que essas primitivas de iterao sejam usadas com mais preciso nos ambientes ES5 se uma implementao do Symbol.iterator estiver presente.

Exemplo: Efeitos no for / of

Sem a flag downlevelIteration ativa, um loop for / of em qualquer objeto sofre um downlevel para um loop for tradicional:

"use strict";
var str = "Ol!";
for (var _i = 0, str_1 = str; _i < str_1.length; _i++) {
    var s = str_1[_i];
    console.log(s);
}Try

Isso geralmente o que as pessoas esperam, mas no 100% compatvel com o comportamento do ECMAScript 6. Certas strings, como emoji (), tm um .length de 2 (ou at mais!), Mas devem iterar como 1 unidade em um loop for-of. Consulte esta postagem no blog de Jonathan New para obter uma explicao mais detalhada.

Quando o downlevelIteration estiver ativado, o TypeScript usar uma funo auxiliar que verifica a implementao do Symbol.iterator (nativo ou polyfill). Se essa implementao estiver ausente, ela retornar iterao baseada em ndice.

Nota: ativar o downlevelIteration no melhora a compatibilidade se o Symbol.iterator no estiver presente no runtime.

Exemplo: Efeitos em Spreads de Arrays

Isso um operador spread em um array:

js
// Cria um novo array onde os elementos so: 1 seguido por todos os elementos do arr2
const arr = [1, ...arr2];

Baseado nas descries, parece fcil fazer um downlevel para ES6:

js
// Mesma coisa, certo?
const arr = [1].concat(arr2);

No entanto, isso claramente diferente em certos casos bem raros. Por exemplo, se o array tiver um buraco no meio, o ndice faltante vai criar uma propriedade prpria quando sofrer o spread, mas isso no acontece quando usamos concat:

js
// Fazemos um array onde temos o elemento do ndice '1' faltando
let faltando = [0, , 1];
let spread = [...faltando];
let concatenado = [].concat(faltando);

// true
"1" in spread;
// false
"1" in concatenado;

Assim como for / of, downlevelIteration vai usar o Symbol.iterator (se presente) para emular de forma mais precisa o comportamento do ES 6.

# Importar Auxiliares - importHelpers

Para algumas operaes de mais baixo nvel, o TypeScript pode usar alguns cdigos auxiliares para operaes como extenso de classes, fazer spread de arrays ou objetos e para operaes async. Por padro esses auxiliares so inseridos em cada arquivo utilizado. Isso pode provocar duplicao de cdigos se o mesmo auxiliar for usado em diferentes mdulos.

Se importHelpers estiver ligado, as funes auxiliares sero importadas do mdulo tslib. Voc precisa ter certeza que o mdulo tslib pode ser importado no runtime. Isso afeta apenas mdulos; Os arquivos de scripts globais no tentaro importar mdulos.

# Mdulos Isolados - isolatedModules

Embora voc possa usar o TypeScript para produzir cdigo JavaScript a partir do cdigo TypeScript, tambm comum usar outros transpilers como Babel para fazer isso. No entanto, outros transpilers operam apenas em um nico arquivo por vez, o que significa que eles no podem aplicar transformaes de cdigo que dependem da compreenso de todo o sistema de tipos. Esta restrio tambm se aplica API ts.transpileModule do TypeScript que usada por algumas ferramentas de construo.

Essas limitaes podem causar problemas de tempo de execuo com alguns recursos do TypeScript, como const enums enamespaces. Definir o sinalizador isolatedModules diz ao TypeScript para avis-lo se voc escrever certo cdigo que no pode ser interpretado corretamente por um processo de transpilao de arquivo nico.

Isso no altera o comportamento do seu cdigo ou, de outra forma, altera o comportamento do processo de verificao e emisso do TypeScript

Alguns exemplos de cdigo que no funcionam quando isolatedModules est habilitado.

Exportaes de identificadores sem valor

No TypeScript, voc pode importar um type e depois export-lo:

import { someType, someFunction } from "someModule";

someFunction();

export { someType, someFunction };Try

Como no h valor para someType, oexport emitido no tentar export-lo (isso seria um erro de tempo de execuo em JavaScript):

js
export { someFunction };

Transpiladores de arquivo nico no sabem se someType produz um valor ou no, ento um erro exportar um nome que se refere apenas a um tipo.

Arquivos Non-Module

Se isolatedModules estiver definido, todos os arquivos de implementao devem ser modules (o que significa que tem alguma forma deimport / export). Ocorre um erro se algum arquivo no for um mdulo:

function fn() {}

Add an import, export, or an empty 'export {}' statement to make it a module.1208'index.ts' cannot be compiled under '--isolatedModules' because it is considered a global script file. Add an import, export, or an empty 'export {}' statement to make it a module.Try

Esta restrio no se aplica a arquivos .d.ts

Referncias a membros const enum

No TypeScript, quando voc faz referncia a um membro const enum, a referncia substituda por seu valor real no JavaScript emitido. Alterando este TypeScript:

declare const enum Numbers {
  Zero = 0,
  One = 1,
}
console.log(Numbers.Zero + Numbers.One);Try

Para este JavaScript:

"use strict";
console.log(0 + 1);Try

Sem o conhecimento dos valores desses membros, outros transpiladores no podem substituir as referncias a Number, o que seria um erro de tempo de execuo se deixado sozinho (uma vez que no h objeto Numbers em tempo de execuo). Por causa disso, quando isolatedModules definido, um erro fazer referncia a um membro ambiente const enum.

  • Default:

    false

# Biblioteca - lib

O TypeScript inclui um conjunto padro de definies de tipo para APIs JS embutidas (como Math), bem como definies de tipo para coisas encontradas em ambientes de navegador (como document). TypeScript tambm inclui APIs para recursos JS mais recentes que correspondem ao target que voc especifica; por exemplo, a definio para Map est disponvel se target for ES6 ou mais recente.

Voc pode querer alter-los por alguns motivos:

  • Seu programa no roda em um navegador, ento voc no quer as definies do tipo "dom"
  • Sua plataforma de tempo de execuo fornece certos objetos de API JavaScript (talvez por meio de polyfills), mas ainda no suporta a sintaxe completa de uma determinada verso ECMAScript
  • Voc tem polyfills ou implementaes nativas para alguns, mas no todos, de uma verso ECMAScript de nvel superior

            O TypeScript 3.8 lanou uma nova estratgia para monitorar as pastas, que crucial para obter as mudanas no node_modules.

            Nos sistemas operacionais como Linux, o Typescript instala monitores de diretrios (ao contrrio dos monitores de arquivos) no node_modules e muitos dos seus sub-diretrios para detectar mudanas nas suas dependncias. Isto devido a quantidade de monitores de arquivos disponveis que eclipsado pelo nmero de arquivos no node_modules, enquanto h muito poucos diretrios para rastrear.

            Como cada projeto deve funcionar melhor com diferentes estratgias, e esta nova abordagem pode no funcionar para o seu fluxo de trabalho, o TypeScript 3.8 introduziu o novo campo watchOptions que permite que o usurio diga ao compilador/servio de linguagem quais estratgias de monitoramento devem ser utilizadas para manter o controle dos diretrios e arquivos.

            # Alternativas na ausncia de observadores - fallbackPolling

            Quando utilizar eventos de arquivos do sistema, essa opo indica as estratgias de verificao que o sistema deve executar quando estiver sem observadores e/ou no suportar os observadores nativos.

            • fixedPollingInterval: Checa por mudanas nos arquivos vrias vezes por segundo a um intervalo pr-determinado.
            • priorityPollingInterval: Checa todos os arquivos por mudanas vrias vezes por segundo, mas usando heursticas para checar alguns tipos de arquivos mais frequentemente que outros.
            • dynamicPriorityPolling: Usa uma fila dinmica onde diretrios que so menos alterados sero checados menos vezes.
            • synchronousWatchDirectory: Desativa a checagem adiada nos diretrios. Adiar a checagem til quando muitas mudanas podem acontecer de uma vez s (ex.: uma mudana em node_modules por executar o npm install), mas voc pode querer desativar isso para configuraes menos comuns.
            • Allowed:

              fixedPollingInterval,
              dynamicPriorityPolling,
              useFsEvents,
              synchronousWatchDirectory

            • Released:

              3.8

            # Observar Diretrio - watchDirectory

            Determina a estratgia que rege como rvores de diretrio so observadas por mudanas em sistemas que no possuem a funcionalidade de observao recursiva de arquivos.

            • fixedPollingInterval: Checa por mudanas nos arquivos vrias vezes por segundo a um intervalo pr-determinado.
            • dynamicPriorityPolling: Usa uma fila dinmica onde diretrios que so menos alterados sero checados menos vezes.
            • useFsEvents (padro): Tenta utilizar os eventos de modificao de diretrios/arquivos do prprio sistema operacional.

            # Observar Arquivo - watchFile

            Define a estratgia como arquivos individuais devem ser observados por mudanas.

            • fixedPollingInterval: Checa todos os arquivos por mudanas vrias vezes por segundo a um intervalo pr-determinado.
            • priorityPollingInterval: Checa todos os arquivos por mudanas vrias vezes por segundo, mas usando heursticas para checar alguns tipos de arquivos mais frequentemente que outros.
            • dynamicPriorityPolling: Usa uma fila dinmica onde arquivos menos modificados so checados menos frequentemente.
            • useFsEvents (padro): Tenta utilizar a funcionalidade nativa de eventos de modificao de arquivos do sistema operacional.
            • useFsEventsOnParentDirectory: Tenta utilizar a implementao de eventos de modificao nativa do sistema operacional para detectar mudanas no diretrio pai de um arquivo.
